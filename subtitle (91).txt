Benvenuto nelle filiali di GitHub. Dopo aver visto questo video, sarai in grado di spiegare lo scopo dei rami in GitHub e descrivere come unire le modifiche in rami. Tutti i file in GitHub sono archiviati su una filiale. Il ramo principale è definitivo, memorizza la versione distribuibile del codice. Il ramo principale viene creato per impostazione predefinita, tuttavia è possibile utilizzare qualsiasi ramo come versione principale , completa e distribuibile del codice. Quando prevedi di cambiare qualcosa, crei un nuovo ramo e gli dai un nome descrittivo. Il nuovo ramo inizia come una copia esatta del ramo originale. Quando apporti modifiche, il ramo che hai creato contiene il codice modificato. Per creare un nuovo ramo, fai clic sul ramo a discesa Principale, aggiungi il nuovo nome del ramo in un nuovo testo e seleziona «Crea ramo». Le branch di GitHub possono essere molto complesse per progetti software di grandi dimensioni. Per un progetto semplice come quelli che stiamo esplorando, considera quanto segue. Inizia con una base comune, la fonte iniziale di questo progetto. A un certo punto, il codice viene ramificato mentre vengono sviluppate nuove funzionalità. In questo esempio, entrambi i rami stanno subendo delle modifiche. Quando i due flussi di lavoro sono pronti per l'unione, il codice di ogni ramo viene identificato come un suggerimento e i due suggerimenti vengono uniti in un terzo ramo combinato. Gli sviluppatori lavorano sui file sorgente di una filiale. Poiché alcuni progetti richiedono un po' di tempo, la fonte non ha subito senso. Per modificare il contenuto di un file, seleziona il file, fai clic sull'icona a forma di matita, apporta le modifiche, confermalo. Quando lo sviluppatore ha completato il lavoro assegnato, esegue il salvataggio del codice per salvare le modifiche. Il commit indica che lo sviluppatore è convinto che il codice rappresenti una piattaforma stabile per la funzionalità o l'insieme di funzionalità in fase di sviluppo. Quando uno sviluppatore inserisce una fonte di modifica nel proprio percorso , deve scrivere un commento che descriva le modifiche. Il commento deve essere significativo e descrittivo. Lo sviluppatore può scegliere di affidarsi alla filiale corrente o crearne una nuova. Infine, fai clic su «Conferma modifiche». Alcune buone pratiche durante la scrittura di un commento : non terminare il messaggio di commento con un punto, ma mantieni i messaggi di commento al di sotto dei 50 caratteri. Utilizza la finestra estesa per i dettagli. Scrivi sempre con voce attiva. Pull viene utilizzato per avviare l'unione dei rami in modo da acquisire le modifiche. Una pull request rende disponibili le modifiche confermate proposte affinché altri possano esaminarle e utilizzarle. Un pull può seguire qualsiasi commit anche se il codice è incompleto. Un pull richiede che un utente approvi le modifiche. Questo può essere l'autore della modifica o può essere assegnato all'interno del team. Nota che GitHub effettua automaticamente una pull request per tuo conto se apporti una modifica su una filiale che non possiedi. Poiché i file di registro sono immutabili, è sempre possibile trovare la persona che approva l'unione della modifica. Per aprire una pull request, fai clic su «Pull request» e seleziona «Nuova pull request». Seleziona il nuovo ramo dalla casella di confronto. Scorri verso il basso per visualizzare le modifiche. Conferma che le modifiche sono quelle che desideri valutare. Aggiungi un titolo e una descrizione alla richiesta. Fai clic su «Crea pull request». L'intento del repository Git è che il ramo principale sia l'unico codice distribuito. Gli sviluppatori possono modificare i file sorgente in un ramo, ma le modifiche non vengono rilasciate finché non vengono salvate. Viene emesso un comando pull, il codice viene esaminato e approvato. Il codice approvato viene reinserito nel codice principale. Per unire una modifica apportata al codice principale, fai clic su «Unisci richiesta pull», quindi su «Conferma unione». Una volta completate tutte le modifiche apportate a un ramo, tale ramo è considerato obsoleto e deve essere eliminato. In questo video, hai appreso che tutti i file in GitHub esistono su un ramo. Il ramo principale contiene la versione distribuibile completa del codice. Crea nuove filiali quando devi modificare il codice. Il nuovo ramo inizia come una copia esatta del ramo originale. Quando apporti modifiche, il ramo che hai creato contiene il codice modificato. È possibile che più di un ramo subisca modifiche contemporaneamente. Le modifiche salvate sono denominate commit. Le pull request consentono ad altri utenti di esaminare e utilizzare le modifiche proposte confermate. Quando sei pronto per unire il codice modificato nel ramo principale, unisci le modifiche apportate al codice principale.
